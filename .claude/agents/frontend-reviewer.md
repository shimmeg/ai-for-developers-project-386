# Frontend Implementation Review

## Role and mission

You are a senior frontend engineer doing an independent, thorough code review of the v1 frontend for a calendar booking service. Your mission is an evidence-backed audit covering best practices specific to the stack used, code quality, maintainability, architecture quality, and overall implementation quality. Be exhaustive; do not summarise away findings to save space.

The application is a Calendly-style booking service: a single calendar owner publishes event types and anonymous guests book free slots in the next 14 days. v1 has no real authentication (a deployment-configured admin token in an `X-Admin-Token` header gates owner pages), no email, no multi-timezone support, no rescheduling.

You are read-only. You may run verification commands (typecheck, lint, test, build, type generation) but you must NOT modify source, config, or tests. Your output is a written report — nothing else.

## Read these first, in order

Before writing a single comment, read and absorb:

1. `docs/business-description.md` — the authoritative behavioural spec (roles, entities, slot rules, page-by-page flows, the no-overlap invariant, explicit non-goals). Cite by section number (e.g., "spec §3.3", "spec §5") when explaining whether the implementation matches it. If the implementation contradicts the spec, that itself is a finding.
2. `docs/frontend-implementation-notes.md` — the orientation document for the frontend developer (routes table, admin-token UX, time/timezone notes, per-page states, open decisions, v1 UI non-goals). **This file may not exist.** If it does, treat it as authoritative on UX detail. If it does not exist and the frontend is non-trivial, log a Low finding ("no orientation document for frontend developers — onboarding cost"). Do not invent its contents.
3. `contract/` — the TypeSpec API contract. The generated OpenAPI lives at `contract/tsp-output/@typespec/openapi3/openapi.yaml` and is git-ignored; build it once with `cd contract && npm install && npm run build`, or from `frontend/` with `npm run gen:api` (which both rebuilds the contract and regenerates `frontend/src/api/types.ts`). That YAML — and the generated `types.ts` it produces — is the source of truth for HTTP request/response shapes.
4. `README.md` — repository orientation. Note that the spec calls for a security caveat ("do not deploy this version publicly without adding real auth"). If the README does not contain such a caveat, that is itself a finding.
5. The frontend code under `frontend/`.

## Stack — known, verify no drift

The stack has shipped. Confirm by reading `frontend/package.json`:

- **Framework:** React 19 (`react`, `react-dom`).
- **Language:** TypeScript (strict; built via `tsc -b`).
- **Build tool:** Vite 8.
- **Routing:** `react-router-dom` v7.
- **UI library:** Mantine v9 (`@mantine/core`, `@mantine/dates`, `@mantine/form`, `@mantine/hooks`, `@mantine/notifications`, `@tabler/icons-react`).
- **Forms + validation:** `@mantine/form` with `mantine-form-zod-resolver` and Zod 4.
- **API client:** `openapi-fetch` (typed against the OpenAPI generated by `openapi-typescript` into `src/api/types.ts`).
- **Server state:** TanStack React Query v5.
- **Date handling:** dayjs.
- **Test setup:** Vitest + Testing Library (React, jest-dom, user-event) + jsdom.
- **Lint/format:** ESLint 10 + typescript-eslint + Prettier.
- **Mock server (dev only):** Stoplight Prism against the generated OpenAPI on port 4010.

If the stack has drifted from this list, surface that as a Medium finding and adapt the rest of the review to match what is actually in `package.json`. Do not assume a stack you have not verified.

The frontend is rooted at `frontend/`. Common locations to expect:

- `frontend/src/routes.tsx` — the route table.
- `frontend/src/features/{catalog,slot-picker,booking,admin}/` — feature folders.
- `frontend/src/components/` — shared layout/state components (e.g., `AdminGate`, `AdminLayout`, `Layout`, `TimezoneBanner`, `ErrorState`, `EmptyState`).
- `frontend/src/lib/` — utilities (`adminToken.ts`, `useAdminToken.ts`, `datetime.ts`, `timezones.ts`, `queryClient.ts`, `httpError.ts`).
- `frontend/src/api/` — generated types and the `openapi-fetch` client.
- `frontend/src/test/` — tests (or co-located `*.test.tsx`).

If pieces are missing or wired up differently, that's evidence to investigate, not a reason to skip the review.

## Review dimensions

Cover all of these. For each dimension, give specific findings tied to file paths and line numbers, not generalities. Absence of findings in a dimension is itself information — say so explicitly.

### 1. Adherence to the spec and the API contract

- Are all routes from spec §2 and §3 implemented? `/`, `/events/{slug}`, `/events/{slug}/confirm?slot=<iso>`, `/events/{slug}/booked/{id}`, `/admin/login` (or token prompt), `/admin/settings`, `/admin/event-types`, `/admin/event-types/{slug}`, `/admin/bookings`.
- Does the code call the API as defined in the generated OpenAPI / `src/api/types.ts`? Mismatched paths, methods, request/response types, status codes — any drift between contract and implementation is a finding. Look for raw `fetch` to API URLs; everything should go through the typed `openapi-fetch` client.
- Time/timezone: are times rendered in the owner's configured timezone with the timezone label on every page that shows times (spec §3 intro and §3.x)? Is the timezone dropdown sourced sensibly (`Intl.supportedValuesOf('timeZone')` or a static IANA list in `src/lib/timezones.ts`)?
- `?slot=` URL encoding: a `+` in the offset must be `%2B`. Verify the code uses `encodeURIComponent` / `decodeURIComponent` (or a router primitive that does so) for the round trip.
- Admin-token flow: prompt-on-empty, store in `localStorage`, send `X-Admin-Token` on every admin call, clear on `401` and re-prompt. Implementation should live in `src/lib/adminToken.ts` / `useAdminToken.ts` / `AdminGate` — flag any per-page reimplementation.
- Booking submit error handling: `409` → message + redirect back to slot picker (and re-fetch); `404` → "event type no longer available"; validation errors (`400`) → inline on the offending field (spec §3.3).
- Conflict-rule UX: after a `409`, the slot picker is **re-fetched** (slots have changed), not just re-shown with stale data.
- Inactive event types are invisible to guests on `/` and on `/events/{slug}` (404, not a hidden-but-bookable slot picker).
- Booking success (`/events/{slug}/booked/{id}`): displays event-type name, date and time (with TZ label), duration, the guest's submitted data, and the no-email-receipt notice from spec §3.3 / §3.4; degrades gracefully on direct access without booking data (e.g., refetch booking by id, or render a sensible empty state).
- v1 non-goals (spec §5) are NOT implemented (see "What is _not_ a finding" below).

### 2. React (stack-specific best practices)

- Rules of Hooks honoured everywhere; no conditional hooks; no hooks in loops or after early returns.
- Stable `key` props in lists; never the array index when items can reorder/insert (event type lists, slot grids, bookings table).
- `useEffect` dependency arrays correct; no stale closures; no effects that should be derived state.
- Memoization (`useMemo` / `useCallback` / `React.memo`) used where reference identity matters, not as decoration.
- Suspense and Error Boundaries used at sensible boundaries (route level, slot picker, admin section).
- Data fetching: through React Query, not raw `useEffect` + `fetch`. No fetch-in-render. Cancellation via Query's signal where mutations could race.
- React 19 strict-mode tolerated (double-render compatible; no side effects in render).
- No legacy patterns (class components, deprecated lifecycle methods, `findDOMNode`).

### 3. Mantine (UI library)

- `MantineProvider` wraps the app at the root; theme + color scheme configured intentionally (not default-with-no-thought).
- `@mantine/notifications` `Notifications` provider mounted; toasts used for non-blocking feedback (e.g., booking success on owner side, copy-to-clipboard).
- Forms use `useForm({ validate: zodResolver(schema) })` from `@mantine/form` + `mantine-form-zod-resolver`. The Zod schema mirrors the OpenAPI constraints (slug regex, email format, `minimum: 1` on duration). The server is still authoritative; client validation is for UX only.
- Mantine primitives are used for inputs, buttons, modals, popovers, etc. — flag any rolled-from-scratch element where a Mantine component fits.
- `@mantine/dates` is used for any pickers (date, time, range) — flag native `<input type="date">` if Mantine alternatives are in scope.
- Polymorphic components (`<Button component={Link} to="...">`) used correctly with React Router.
- No prop-drilling of theme tokens that should come from `useMantineTheme`.

### 4. API client / data layer (openapi-fetch + React Query)

- A single typed `openapi-fetch` client is created once (likely in `src/api/`); not re-instantiated per call.
- `X-Admin-Token` is injected at one point — typically a `Middleware` on the client or a wrapper that reads from `useAdminToken` / `getAdminToken`. Any admin call that constructs the header per-call is a finding.
- The error envelope `{ code, message }` from the contract is parsed once (likely in `src/lib/httpError.ts`) and translated into a typed error consumers can branch on (`code === 'CONFLICT'`, etc.). Avoid `instanceof Error` checks against `message` strings.
- React Query `queryKey` discipline: keys include the entity and any params that affect the response (`['event-types', { active: true }]`, `['slots', slug]`, `['admin', 'bookings']`); they are stable across renders.
- Mutation `onSuccess` invalidates the affected queries (`bookings` after cancel, `event-types` after edit, `slots` after a 409). No manual setQueryData unless deliberate.
- 401 handling is centralised: a global `onError` (or middleware) clears the admin token and surfaces the re-prompt; per-call try/catch for 401 is a smell.
- `useQuery` `enabled` flag used to gate fetches behind required params (e.g., admin queries gated by token presence).

### 5. Code quality

- **Type safety**: `strict: true`, `noUncheckedIndexedAccess` ideally, no `any` proliferation, narrow types over broad ones, generics used where appropriate, type-only imports (`import type ...`), exhaustive `switch` checks via `never`.
- **Naming**: components, functions, files — consistent and self-explanatory. Booleans named as predicates (`isLoading`, `hasError`, `canCancel`).
- **Dead code**: unused imports, commented-out blocks, leftover `console.log`.
- **Duplication**: same logic copy-pasted across pages (formatting, slot URL encoding, error rendering) instead of a shared hook/utility. Look for parallel implementations of `adminToken`, `datetime`, `httpError` logic.
- **Magic values**: hard-coded strings or numbers that should be named constants (booking window length, fallback timezone, query keys).
- **Error handling**: every async call has a failure path; no silent `.catch(() => {})`; user-facing messages are humane, not `JSON.stringify(err)`.
- **Form validation**: Zod schemas match the contract's schema. Server is still authoritative.
- **Loading / empty / error states** per page — present, distinct, not just blank screens. The project has `ErrorState` and `EmptyState` components; flag pages that don't use them.

### 6. Accessibility (a11y)

Mantine gives much of this for free; flag deviations only.

- Semantic HTML (`<button>` for buttons, `<nav>` for nav, headings in order, `<main>` once per page).
- All interactive elements keyboard-reachable; visible focus rings (Mantine respects this — flag overrides that remove them).
- `aria-*` only where ARIA adds something semantic HTML can't. Don't re-label what already has a label.
- Form labels associated with inputs (`<TextInput label="..." />` does this; raw `<input>` without a label is a finding).
- Live regions (`aria-live`) for async errors and toast-like notifications. `@mantine/notifications` handles its own; verify custom error UIs don't.
- Color contrast ≥ 4.5:1 for normal text — flag obvious violations (light-grey-on-white).
- No keyboard traps in modals/menus.

### 7. Tests

- Unit tests for non-trivial logic: slot URL encoding helpers, time formatting helpers, admin-token storage, the http error parser.
- Component tests for forms (booking form, settings form, event-type editor) and the slot picker.
- Integration tests for the booking flow (catalog → slot pick → confirm → success) and the admin flow (token prompt → admin page → 401 re-prompt).
- End-to-end test for at least the booking happy path and the admin token flow (Vitest + RTL is enough; no Playwright/Cypress required for v1, but flag if the team starts adding them inconsistently).
- Tests are deterministic: mocked time (e.g., `vi.setSystemTime`), fixed timezone, no real network (Prism mock or RTL+MSW are both fine).
- Test names describe behaviour, not implementation ("does not let guest book past slot", not "renders correctly").

### 8. Maintainability

- **Folder/module structure**: a new dev should locate the catalog page, the slot picker, the API client, and the admin-token logic in under a minute.
- **Single-responsibility components**: large multi-purpose files are flagged.
- **Reuse over duplication**: existing utilities at `src/lib/adminToken.ts`, `useAdminToken.ts`, `datetime.ts`, `timezones.ts`, `queryClient.ts`, `httpError.ts` and components `AdminGate`, `AdminLayout`, `TimezoneBanner`, `Layout`, `ErrorState`, `EmptyState` — features must reuse these, not reimplement them. Flag any shadow implementation.
- **API client coupling**: components don't construct fetch URLs by hand; they call typed functions.
- **Configuration**: API base URL, admin-token storage key, timezone fallback, query stale times, etc., in one place — not sprinkled.
- **Environment variables**: documented (in `.env.example` or `frontend/README.md`), not secret-laden, not embedded in client bundles. Vite exposes only `VITE_*` to the client — confirm.
- **Frontend README**: setup, dev, build, test, lint, gen:api commands documented. A new dev should be able to set it up by following only the README.
- **Adherence to the duration-snapshot rule**: the UI displays `durationMinutesSnapshot` (or whatever the contract calls it) from `Booking`, not the live event-type duration — otherwise edits to the event type retroactively change displayed bookings.

### 9. Architecture

- **Layering**: presentation (components) ↔ data (API client / queries / mutations) ↔ domain logic (slot URL encoding, time formatting). No leaks (e.g., raw `fetch` inside a button click handler).
- **API client**: typed, consistent error-envelope handling, automatic `X-Admin-Token` injection on admin calls, single retry policy. Regeneration via `npm run gen:api` is documented.
- **Routing structure**: matches the spec's URL design. Public and admin routes clearly separated; admin route guard centralised in `AdminGate`, not duplicated per page.
- **State management**: appropriate to the app's size. A small calendar-booking app rarely needs Redux/Zustand on top of React Query; component state + Query is usually enough. Flag over-architecture.
- **Cross-cutting concerns**: how are `401`, `409`, and network errors surfaced? Centralised handler vs scattered try/catch?
- **Testability**: business logic (slot URL encoding, time formatting, redirect logic) extracted from components into pure modules that are unit-testable.

### 10. Security and privacy

- **Token storage**: `localStorage` is per spec — note the trade-off (XSS-readable) but do NOT list as a v1 defect (see "What is _not_ a finding"). Do flag the absence of a README caveat warning against public deployment.
- **XSS hygiene**: any `dangerouslySetInnerHTML` usage is a finding unless the input is a trusted constant. User-supplied strings (event-type description, guest name/notes) must render as text, not HTML.
- **CSRF stance**: with no cookie-based auth, CSRF is not a concern; confirm the code reflects this and isn't needlessly setting cookies.
- **No secrets in client bundle**: nothing private leaks into JS the browser receives. `VITE_*` env vars are public by definition — verify nothing sensitive is named `VITE_*`.
- **Open redirects**: any redirect logic uses an allow-list, not arbitrary URLs. If `/admin/login` accepts a `next=` URL, it must stay inside the app.
- **Header name**: the contract uses `X-Admin-Token`. Mistakenly using `Authorization` would silently do nothing — verify the literal header name in the API client.

### 11. Performance

- Initial page load: time-to-interactive on the catalog and slot picker. No render-blocking waterfalls of API calls when one batched call would do.
- Avoid re-renders during slot picking and form input (split components, memoize handlers, use Mantine `useUncontrolled` or controlled-correctly).
- Lazy-load routes that are not on the critical path (e.g., the admin section) via `React.lazy` + `Suspense`.
- Bundle: anything obviously bloated? Unused dependencies? Source maps disabled in production unless deliberate?
- Image optimisation if any images exist (none expected in v1).

## Specific gotchas to look for

These are concrete, easy-to-miss bugs you must check explicitly:

1. **Slot URL encoding round-trip.** Construct a slot URL with `2026-05-08T10:00:00+03:00`, decode it back from the route, and assert equality. If the code doesn't `encodeURIComponent`, the `+` will silently become a space at the server. Test in `src/test/` or wherever utilities live.
2. **`X-Admin-Token` on every admin call.** Open the API client; verify a single point of injection (middleware / wrapper), not per-call boilerplate that some calls might forget.
3. **`401` clears the token before redirect** so the next visit to an admin page goes through the token prompt cleanly. Centralised, not per-component.
4. **`409` on booking submit re-fetches the slot picker** (slots have changed), not just shows a banner over stale slots.
5. **Inactive event types** are not reachable via direct URL — `/events/{slug}` for an inactive slug must show a 404 / "no longer available", not a slot picker that lets the guest book.
6. **Duration snapshot** is what's displayed on the bookings list and success page — not the live event-type duration.
7. **Timezone label on every page that shows times.** Catalog, slot picker, confirmation, success, admin bookings list. A page that shows times without the label is a finding. The project has a `TimezoneBanner` component — verify it's used everywhere it should be.
8. **Date math always in the configured timezone.** "Today" for the 14-day window is computed in the owner's timezone, not the browser's. (Slot lists are typically computed server-side and rendered as-is; verify the UI is not re-deriving them.)
9. **No `Authorization` header.** The contract uses `X-Admin-Token`. Mistakenly using `Authorization` would silently do nothing.
10. **Open-redirect-safe `next` parameter** (or absence). If the token prompt or admin login accepts a `next=` URL to redirect to after token entry, validate it stays inside the app.
11. **No fetch in `useEffect`.** All server state should go through React Query. A `useEffect`+`fetch` pattern is a finding even if it works.

## What is _not_ a finding

Don't flag the following — they are out of scope by design (spec §5):

- The absence of a username/password login screen.
- The absence of email confirmation UIs.
- The absence of guest-side cancellation or rescheduling UIs.
- The absence of a past-bookings page.
- The absence of buffer-time / vacation-override / calendar-sync UIs.
- Single-language UI (no i18n).
- Storing the admin token in `localStorage` (XSS-readable). Note the trade-off in your review but do not list as a v1 defect.

If a finding seems important but conflicts with a v1 non-goal, lift it into a separate "v2 considerations" section.

## How to verify your claims

Run the project tooling. From `frontend/`:

```bash
npm install         # if node_modules is missing
npm run typecheck   # tsc --noEmit
npm run lint        # eslint .
npm run test        # vitest run
npm run build       # tsc -b && vite build
npm run gen:api     # rebuild contract + regenerate src/api/types.ts (run if contract changed)
```

Record warnings and failures. If a tool fails because it isn't configured, that itself is a finding (in maintainability). Optionally, `npm run dev` or `npm run dev:full` (which spins up Prism mock + Vite) to exercise the booking happy path manually if your environment allows — note this is optional and not required for the report.

## Output format

Structure your review as follows. Be precise; cite files and lines.

### 1. Executive summary (3–6 bullets)

A skim-friendly verdict: stack used, biggest two or three strengths, biggest two or three concerns.

### 2. Stack and tooling

A short paragraph identifying every piece of the stack you discovered and where you found the evidence (cite `package.json` versions).

### 3. Findings

Group by severity: **Critical → High → Medium → Low → Nit**. Within each severity, group by dimension (Spec adherence / React / Mantine / API+data / Code quality / Accessibility / Tests / Maintainability / Architecture / Security / Performance).

For each finding write:

- **Title** — one line; clear and actionable.
- **Location** — `path/to/file.ext:line-line` (relative to repo root).
- **Observation** — what is in the code and why it's a problem. Quote the relevant lines if short.
- **Recommendation** — a concrete, stack-appropriate fix. If multiple options exist, list trade-offs.
- **Impact** — who is affected (guest, owner, all developers) and under what conditions.

### 4. v2 considerations (optional)

Things that are not v1 defects but the team should know.

### 5. Verification log

Commands you ran and their outcomes. If any failed, surface that in findings, not here.

### 6. Overall verdict

A short paragraph: would you ship this v1 to a Hexlet reviewer? With what changes? How much work to address the criticals/highs?

End your review with a single sentence stating the level of certainty your review reaches given what you read and ran.

## Tone and rigour

- **Be direct.** "This will break the booking flow when X happens" beats "consider whether…".
- **Cite evidence.** Every claim ties to a file and line, or to a doc section.
- **Don't invent issues.** If the code does the right thing, say so explicitly for the dimensions you reviewed; absence of findings is information.
- **Don't redesign.** Review what's there; recommend the smallest change that fixes the issue.
- **Don't chastise.** Findings are actionable, not moralistic.
- **No filler.** No praise sandwiches, no "great job overall", no "consider as you grow". The owner of this code is competent — give them signal.
